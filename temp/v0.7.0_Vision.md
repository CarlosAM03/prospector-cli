# v0.7.0 — Search Automation Vision

## Purpose

The objective of v0.7.0 is **not** to extract new types of information.

The Website Engine, enrichment pipeline, exporters and normalized models are already capable of producing structured business data.

The objective of this version is to make the **search itself** reliable, complete and reusable.

This release transforms the Google Maps scraper from a proof of concept into a robust extraction engine capable of processing an entire search from the user's query to the final exported file.

---

# Functional Objective

From the user's perspective, nothing changes in the interface.

The execution flow remains exactly the same.

```
Search

↓

Extraction

↓

Website Enrichment

↓

Export
```

The difference is what happens behind the scenes.

Instead of extracting only the businesses initially rendered by Google Maps, the engine will automatically continue navigating until no additional results remain available.

The user should perceive a single improvement:

> Every available result is processed automatically.

If Google Maps contains:

* 8 businesses
* 50 businesses
* 120 businesses
* 300 businesses

the engine should continue working until Google Maps reports that no additional content can be loaded.

The exported file should therefore represent the complete search instead of only the initially visible portion.

---

# User Experience

At the end of v0.7.0 the user should be able to execute searches such as:

```
maquila tijuana

hospital mexicali

hotel ensenada

coffee san diego

industrial park baja california

transportes rosarito

consultorio veterinario tecate
```

without needing to adapt the wording to a predefined syntax.

The search query becomes user input instead of engine input.

The engine is responsible for adapting itself to Google Maps, not the opposite.

---

# Expected Functional Behavior

The user should be able to:

* Enter any search query.
* Execute the search.
* Wait until the engine finishes automatically.
* Receive the selected export format.

The engine should determine internally:

* when Google Maps is still loading,
* when additional scrolling is required,
* when new businesses have appeared,
* when extraction should continue,
* when the end of the result list has been reached.

The user should never need to know how many scrolling operations occurred internally.

---

# What Changes Internally

Although the external workflow remains identical, the internal execution pipeline changes significantly.

Current execution:

```
Search

↓

Wait for feed

↓

Extract visible businesses

↓

Export
```

Target execution:

```
Search

↓

Synchronization Engine

↓

LazyCharge Engine

↓

Business Extraction

↓

Detail Extraction

↓

Website Inspection

↓

Export
```

The scraper no longer depends on a fixed initial DOM state.

Instead, reusable synchronization engines coordinate the browser state before extraction begins.

This follows the architectural principle that reusable execution behavior belongs in engines rather than scraper implementations.

---

# Why LazyCharge Engine Exists

The LazyCharge Engine is **not** a Google Maps feature.

It is the first reusable navigation engine introduced into Prospector CLI.

Google Maps is simply the first scraper that requires this capability.

Its responsibility is to automate progressive content loading whenever a public source reveals information incrementally.

For Google Maps this means:

* detecting when more results exist,
* performing automatic scrolling,
* waiting until new content appears,
* repeating the process until the source reaches its natural end.

Future scrapers may reuse the same engine without modifying its implementation.

---

# Stability Objective

v0.7.0 also changes the reliability expectations of the engine.

Current behavior:

* Successful execution depends on Google presenting the expected initial DOM.
* Missing selectors may abort the entire extraction.
* Different searches may behave inconsistently.

Expected behavior:

The engine should continue operating even when:

* Google delays rendering,
* the initial page differs,
* selectors appear later,
* searches return very large result sets,
* no additional results exist,
* recoverable synchronization issues occur.

Recoverable situations should be handled internally whenever possible instead of terminating the execution.

Search completion becomes the normal outcome.

Unexpected termination becomes the exception.

---

# What This Version Does NOT Change

This release does not modify:

* Business model
* Website Engine
* Metadata extraction
* Email extraction
* Language detection
* Export formats
* CLI interaction
* Output structure

The information extracted remains the same.

The quality of navigation is what changes.

---

# Success Criteria

v0.7.0 can be considered complete when the following statements are true.

* Any valid search query can be executed without requiring predefined wording.
* The engine automatically traverses the complete Google Maps result list.
* Extraction continues until Google Maps has no additional results to load.
* Synchronization is based on browser state instead of fixed assumptions.
* Recoverable search failures no longer terminate the pipeline with LookupError.
* The execution reaches the exporter regardless of the number of scrolling operations required.
* The user receives the selected output format after the complete search finishes.

---

# Version Summary

The previous releases focused on answering:

> What information can Prospector CLI extract?

v0.7.0 answers a different question:

> How can Prospector CLI reliably reach every available result before extracting it?

This release establishes the first reusable navigation infrastructure of the project.

It is the foundation upon which future scrapers will automate dynamic websites while preserving the same public execution pipeline and architectural principles.
